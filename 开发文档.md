# Ariadne 开发文档（详细版，基于 user_stories_100）

## 1. 文档定位与适用范围
本开发文档用于把 `user_stories_100.md` 转化为可执行工程计划，面向：
- 产品经理：范围、优先级、里程碑、验收标准。
- 研发负责人：任务拆解、跨模块依赖、人力配比、风险控制。
- 前后端工程师：接口契约、数据结构、联调路径、测试基线。
- QA：测试矩阵、验收用例、回归策略。

适用范围：
- 覆盖 E1~E10 全部 Epic。
- 落地优先级以 P0/P1 为主线，P2/P3 给出设计预留与交付顺序。

## 2. 研发目标与成功标准

### 2.1 北极星目标
- 在未登录、本地优先的前提下，实现“输入主题 -> 生成课件 -> 分 chunk 学习 -> 即时提问 -> 导出离线”的稳定闭环。

### 2.2 量化成功标准
- 功能可用性：P0 用户故事 100% 达标；P1 用户故事 >= 85% 达标。
- 性能：常规主题首屏生成结果 <= 15s（P95 <= 22s）。
- 稳定性：生成主链路成功率 >= 97%；导出成功率 >= 99%。
- 质量：核心后端单测覆盖率 >= 70%，主链路 E2E 全绿。

## 4. 分阶段开发计划（建议 3 个 Program Increment）

## 4.1 PI-1（第 1~6 周）核心闭环
目标：交付 P0 主链路。

周计划：
- 第 1 周：需求冻结、交互稿冻结、接口草案评审。
- 第 2 周：主题生成页 + 后端生成编排 MVP。
- 第 3 周：chunk 阅读页 + “聊这个点” + 上下文问答。
- 第 4 周：上传资料与检索接入 + 进度流。
- 第 5 周：导出 HTML + 未登录模式 + 日志埋点。
- 第 6 周：联调、性能压测、缺陷收敛、发布演练。

退出标准：
- P0 用户故事全部通过验收。
- 无 P0/P1 阻断缺陷。
- E2E 主链路连续 3 次回归通过。

## 4.2 PI-2（第 7~12 周）可用性增强
目标：交付主要 P1 能力。

周计划：
- 第 7~8 周：参数恢复、失败重试、多轮会话、选中文本提问。
- 第 9~10 周：来源治理（偏好、白名单、解析状态、引用展示）。
- 第 11 周：个性化画像与缺口识别、跨页编辑计划。
- 第 12 周：质量与性能专项（单测覆盖、趋势监控、E2E 完整性）。

退出标准：
- P1 目标故事通过率 >= 85%。
- 后端核心模块单测覆盖 >= 70%。
- 生成性能达到目标阈值。

## 4.3 PI-3（第 13~22 周）进阶学习系统
目标：P2/P3 高价值能力上线。

交付方向：
- 图谱、复习调度、成果页、版本标签、导入导出增强。
- 可访问性与国际化完善。

## 5. 模块化研发拆分

## 5.1 前端模块与交付件
- `app-shell`：路由、布局、侧边栏、全局状态。
- `topic-generator`：主题输入、参数面板、生成进度、失败重试。
- `courseware-reader`：chunk 导航、折叠、学习进度、笔记、收藏。
- `ai-sidebar`：上下文聊天、多轮追问、改写草稿、应用与撤销。
- `source-center`：来源显示、检索偏好、白名单管理、空结果建议。
- `asset-export`：上传列表、解析状态、HTML/ZIP 导出。
- `offline-pwa`：离线缓存、恢复提示、同步补偿。
- `i18n-a11y`：中英切换、键盘焦点、读屏文本。

每个模块产出：
- 页面与组件。
- 状态管理与请求层。
- 单元/集成测试。
- 埋点与错误上报。

## 5.2 后端模块与交付件
- `gateway`：统一路由、限流、参数校验、trace_id 注入。
- `generation-orchestrator`：检索/生成/组装流水线。
- `retrieval-service`：网络检索、本地索引、来源评分。
- `qa-service`：chunk 上下文问答与补充建议。
- `courseware-service`：课件、chunk、标签、版本。
- `asset-service`：上传、解析、分块、索引。
- `export-service`：HTML/ZIP/只读版导出。
- `profile-service`：学习画像、个性化参数、本地模式。
- `observability-service`：日志、指标、告警、趋势看板。

每个模块产出：
- API + 领域逻辑 + 数据访问层。
- 模块级单元测试。
- 合同测试（Contract Test）。

## 6. Epic 级详细实现说明

## 6.1 E1 主题输入与课件生成
对应故事：US001/002/003/005/006/007/008/009/010。

前端细节：
- 表单字段：`topic`、`keywords[]`、`difficulty`、`style`、`template`。
- 交互约束：topic 非空且长度 2~120；keywords 最多 10 个。
- 状态机：`idle -> validating -> queued -> retrieving -> generating -> assembling -> done/failed`。
- 参数恢复：localStorage 按用户会话保存；支持“清空历史参数”。

后端细节：
- 生成请求接收参数后，创建 `generation_job`。
- 编排阶段结果实时写入 `generation_events`。
- 失败重试沿用原参数并复制上下文快照，避免重复召回不一致。

验收补充：
- 10 秒内必须出现进度状态。
- 首屏结果包含至少 3 个 chunk。
- 导出 HTML 可离线打开且样式、目录、代码块完整。

## 6.2 E2 Chunk 交互学习体验
对应故事：US011~US020。

前端细节：
- 侧边栏目录支持关键字过滤、当前 chunk 高亮。
- 每个 chunk 有独立状态：`expanded/collapsed`、`understood/not_understood`、`favorite`。
- 学习进度条计算：`visited_or_understood_chunk_count / total_chunk_count`。
- 文本选择提问：自动采集 selection 文本 + offsets + chunk_id。

后端细节：
- chunk 状态写入 `chunk_state` 表，按 `courseware_id + chunk_id` 维度更新。
- 简版/深入版回答使用不同 prompt profile，输出结构统一。

验收补充：
- 跳转时延 <= 1 秒（前端定位 + 滚动完成）。
- 刷新后笔记可恢复。

## 6.3 E3 追问、补充与关联课件
对应故事：US021~US030。

实现细节：
- “我没懂”入口默认携带 chunk 上下文与最近 3 轮会话摘要。
- AI 补充默认先生成草稿，用户必须显式采纳。
- 关联 chunk 在数据结构中记录 `parent_chunk_id` 与 `relation_type=expansion`。
- 版本记录存储变更摘要和 diff 元数据，支持后续对比。

验收补充：
- 回答中必须包含关联 chunk 编号和术语引用。
- 拒绝补充后正文不产生版本变更。

## 6.4 E4 多源检索与来源管理
对应故事：US031~US040。

实现细节：
- 检索开关 OFF：仅使用本地知识与内置基础知识。
- 来源偏好权重：`doc/blog/paper` 默认 `0.4/0.3/0.3`，可配置。
- 白名单命中策略：白名单开启时，结果域名必须 100% 命中。
- 上传链路：`upload -> parse -> chunk -> index -> ready`。
- 空结果建议：返回 3 条改写建议并可一键重试。

验收补充：
- 解析状态每 3 秒刷新。
- 每个 chunk 至少展示 1 条来源（如有可用来源）。

## 6.5 E5 学习记忆与个性化讲解
对应故事：US041~US050。

实现细节：
- 画像字段：学习目标、基础水平、偏好类比、已掌握概念。
- 生成前执行 `prerequisite_gap_check`，输出缺口列表。
- 本地模式：`profile.local_only=true` 时禁发远程同步请求。

验收补充：
- 个性化开启后，至少 1 个 chunk 显示“关联已学”。

## 6.6 E6 知识图谱与星球可视化（PI-3）
对应故事：US051~US060。

实现细节：
- 概念抽取后写入 `concept_nodes`，关系写入 `concept_edges`。
- 2D 图谱渲染优先保证 100 节点内流畅。
- 节点点击需精准回跳 chunk。

## 6.7 E7 AI 侧边栏对话与智能编辑
对应故事：US061~US070。

实现细节：
- 聊天请求统一附带 `page_id/chunk_id/selection/context_window`。
- “改写当前 chunk”返回 draft，不直接覆盖正文。
- 跨页修改必须先展示影响面清单并逐项勾选。
- 撤销机制：只允许撤销最近一次已应用变更。

验收补充：
- 侧边栏可拖拽宽度并保留会话内状态。
- “继续展开”可保持上下文连续性。

## 6.8 E8 复习调度与学习路径（PI-3）
对应故事：US071~US080。

实现细节：
- chunk 转卡片时保留来源、难度、章节路径。
- 间隔重复算法参数可配置并可视化。

## 6.9 E9 课件资产与成果输出
对应故事：US081~US090。

实现细节：
- 完整包导出：`index.html + assets/* + manifest.json`。
- 全局搜索支持关键词高亮与 chunk 精确定位。
- 版本标签遵循 semver（如 `v1.2.0`）。

## 6.10 E10 平台基础能力与质量保障
对应故事：US091~US100。

实现细节：
- 未登录模式默认启用本地存储。
- 离线模式下可打开已缓存课件、笔记、聊天历史。
- 维护者看板展示 7 天错误趋势与模块分布。
- i18n 按 key 管理，避免硬编码文案。

## 7. API 详细契约（P0/P1 核心）

## 7.1 生成课件
`POST /api/coursewares/generate`

请求体：
```json
{
  "topic": "Transformer",
  "keywords": ["attention", "positional encoding"],
  "difficulty": "beginner",
  "style": "engineering",
  "template": "tutorial",
  "retrieval": {
    "web_enabled": true,
    "source_weight": {"doc": 0.5, "blog": 0.2, "paper": 0.3},
    "domain_whitelist": ["arxiv.org"],
    "domain_blacklist": []
  },
  "profile_id": "pf_123"
}
```

响应：
```json
{
  "job_id": "job_001",
  "courseware_id": "cw_001",
  "status": "queued",
  "trace_id": "tr_abc"
}
```

## 7.2 生成进度
`GET /api/coursewares/{id}/progress`

响应：
```json
{
  "courseware_id": "cw_001",
  "phase": "generating",
  "progress": 62,
  "events": [
    {"ts": "2026-03-02T14:20:11Z", "phase": "retrieving", "message": "retrieval finished"},
    {"ts": "2026-03-02T14:20:18Z", "phase": "generating", "message": "drafting chunks"}
  ]
}
```

## 7.3 chunk 问答
`POST /api/chunks/{id}/ask`

请求体：
```json
{
  "question": "这段注意力公式我没懂",
  "page_id": "pg_12",
  "chunk_id": "ck_88",
  "selection": {
    "text": "Attention(Q,K,V)=softmax(QK^T/sqrt(dk))V",
    "start": 120,
    "end": 168
  },
  "mode": "deep"
}
```

响应：
```json
{
  "answer": "...",
  "linked_chunk_id": "ck_88",
  "sources": [{"title": "Attention Is All You Need", "url": "https://arxiv.org/..."}],
  "next_suggestions": ["要不要我先讲向量维度含义？"]
}
```

## 7.4 上传资料
`POST /api/assets/upload`（multipart）

响应：
```json
{
  "asset_id": "as_01",
  "status": "queued"
}
```

## 8. 数据模型（字段级）

- `status` enum(`draft`,`generating`,`ready`,`failed`,`archived`)

## 9. 研发流程与协作机制

流程定义：
- PRD 冻结 -> 技术方案评审 -> 接口评审 -> 开发 -> 联调 -> QA -> 灰度 -> 全量。

门禁规则：
- 合并前必须通过 lint、unit test、contract test。
- P0/P1 接口变更必须更新 API 文档与联调 mock。
- 每个 Story 合并时必须挂接对应 US 编号。

分支策略：
- `main`：稳定分支。
- `release/*`：发布候选分支。
- `feature/USxxx-*`：故事分支。

## 10. 测试矩阵（详细）

## 10.1 功能测试矩阵（示例）
- 用例 `TC-E1-001`：空 topic 提交，按钮禁用。
- 用例 `TC-E1-002`：topic 有效提交，10 秒内出现进度。
- 用例 `TC-E2-004`：点击 chunk 目录，1 秒内定位。
- 用例 `TC-E3-003`：拒绝 AI 补充后正文版本不变化。
- 用例 `TC-E4-005`：白名单只含 `arxiv.org` 时，结果域名必须全命中。
- 用例 `TC-E7-006`：改写草稿未确认前不落库。
- 用例 `TC-E10-007`：离线状态打开已缓存课件成功。

## 10.2 性能测试
- 场景 A：标准主题 + web 检索开，100 并发请求。
- 场景 B：仅本地检索，100 并发请求。
- 场景 C：导出 50MB 课件包。

指标阈值：
- 生成首屏 P95 <= 22s。
- 问答响应 P95 <= 5s。
- 导出任务成功率 >= 99%。

## 10.3 稳定性测试
- 长稳压测 24 小时，观察错误率、内存增长、队列堆积。
- 故障注入：LLM 超时、检索空返回、索引服务不可用。

## 11. 可观测性与运维细则

日志规范：
- 所有请求挂接 `trace_id`。
- 关键动作记录 `event_type`、`entity_id`、`actor`、`duration_ms`。

指标规范：
- 生成：`gen_latency_ms`、`gen_success_rate`、`gen_retry_count`。
- 检索：`retrieval_hit_rate`、`source_credibility_distribution`。
- 对话：`qa_latency_ms`、`qa_context_hit_rate`。
- 导出：`export_latency_ms`、`export_fail_rate`。

告警策略：
- 5 分钟失败率 > 5% 告警。
- 生成 P95 > 30s 连续 10 分钟告警。
- 解析任务堆积 > 100 告警。

## 12. 风险清单与缓解方案
- 风险：模型回答漂移导致 chunk 结构不一致。
- 缓解：结构化输出 schema + 后处理修复器 + 回退模板。
- 风险：上传文档质量参差导致检索污染。
- 缓解：解析评分 + 低质量片段降权 + 可视化来源。
- 风险：多版本编辑冲突。
- 缓解：乐观锁 + 最近一次修改撤销 + 自动快照。
- 风险：离线缓存膨胀。
- 缓解：LRU 清理策略 + 用户可见存储占用管理。

## 13. 上线与回滚

发布流程：
- Canary（5%）-> 小流量（30%）-> 全量（100%）。
- 每阶段至少观察 2 小时关键指标。

回滚流程：
- 应用层回滚到前一 release tag。
- 数据层使用版本快照恢复课件。
- 对用户保留当前版本副本，防止数据丢失。

## 14. Story 到任务包映射（执行模板）
每条用户故事在 Jira/TAPD 中拆为至少 4 类任务：
- `FE` 页面/交互实现。
- `BE` 接口/逻辑实现。
- `QA` 用例与自动化补充。
- `OPS` 监控与告警配置。

示例（US007）：
- FE：进度组件 + SSE 订阅。
- BE：阶段事件上报。
- QA：阶段切换时序测试。
- OPS：生成阶段延迟指标看板。
